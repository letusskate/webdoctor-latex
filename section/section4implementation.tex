\section{Implementation}

\subsection{Dataset Construction: IHDPD}

We construct the \textbf{Internet Hospital Doctor-Patient Dialogue (IHDPD)} dataset from the consultation records of a real-world internet hospital platform. The data collection and processing pipeline adheres to strict ethical and privacy protocols approved by the First Hospital of Hebei Medical University.

\textbf{Data Source.} The raw data consist of text-based consultation records collected over a 24-month period (January 2025 to December 2026) from an operational internet hospital platform serving over 100,000 registered patients and more than 500 licensed physicians across 15 clinical departments, including internal medicine, cardiology, dermatology, pediatrics, obstetrics and gynecology, orthopedics, neurology, gastroenterology, endocrinology, psychiatry, ophthalmology, otolaryngology, urology, respiratory medicine, and general surgery. After de-identification and quality filtering, the training set comprises 20,149 dialogue windows ($\sim$216,000 individual messages) and the test set comprises 2,059 dialogue windows ($\sim$22,000 individual messages), spanning 766 licensed physicians across 121 clinical sub-departments (aggregated into 15 broad categories).

\textbf{De-identification.} All personally identifiable information (PII) and protected health information (PHI) are systematically removed using a hybrid approach combining rule-based pattern matching (for structured identifiers such as names, ID numbers, phone numbers, and addresses) with a fine-tuned named entity recognition model~\cite{johnsonMIMICIIIFreelyAccessible2016} (for contextual identifiers such as hospital names, dates, and locations). The de-identification pipeline achieves a recall of 98.7\% and a precision of 97.2\% on a manually annotated validation set of 500 dialogues. All de-identified records undergo a secondary manual review by two independent annotators to ensure completeness.

\textbf{Dialogue Structure.} Each consultation session in IHDPD is structured as a multi-turn dialogue between one patient and one physician. A typical session comprises an initial symptom description by the patient, followed by one or more rounds of physician inquiry, diagnostic reasoning, and treatment recommendation. The dataset preserves the natural turn-taking structure, including timestamps (with dates generalized to month-level granularity), department labels, and physician specialty annotations.

\textbf{Data Partitioning.} For each physician, dialogue windows are partitioned chronologically into training (70\%, earliest consultations), validation (15\%), and test (15\%, most recent consultations) sets. The training set comprises 20,149 dialogue windows ($\sim$216,000 individual messages) from 415 physicians, while the held-out test set contains 2,059 windows ($\sim$22,000 messages) from 351 physicians. All experiments use random seed 42 for data splitting, model initialization, and training order to ensure reproducibility.


\textbf{Hyperparameter Selection.} LoRA rank $r$ and SCF trim fraction $\tau$ were selected via grid search over $r \in \{4,8,16,32\}$ and $\tau \in \{0.10,0.15,0.20,0.25,0.30,0.40\}$ on the validation set. The optimal configuration ($r=8$, $\tau=0.20$) was then evaluated on the held-out test set.

\subsection{System Architecture}

Our system is implemented as a modular pipeline comprising four stages:

\textbf{Stage 1: Data Ingestion and Preprocessing.} Raw consultation records are streamed through the de-identification pipeline, tokenized using the base model's tokenizer, and formatted into instruction-following templates that include the patient query, physician department, and expected response format. The preprocessing stage handles batching, shuffling, and dynamic padding using standard data utilities~\cite{zheng2024llamafactory}.

\textbf{Stage 2: LoRA Training.} We perform LoRA fine-tuning using LLaMA-Factory~\cite{zheng2024llamafactory} (via WeClone~\cite{xmingXming521WeClone2026}) with the following hyperparameters: LoRA rank $r = 8$, LoRA alpha $\alpha = 16$, dropout rate $0.1$, target modules $\{q\_proj, v\_proj, k\_proj, o\_proj\}$, learning rate $2 \times 10^{-4}$ with cosine scheduling, batch size 4 with gradient accumulation over 8 steps (effective batch size 32), and a maximum of 10 training epochs with early stopping (patience = 3). Training is conducted on 8$\times$ NVIDIA H20-3e (141GB) GPUs using DeepSpeed ZeRO Stage-2 optimization.

\textbf{Stage 3: DAR Router.} The router encodes physician style descriptions $s_k$ using a distilled Sentence-BERT model~\cite{reimers2019sentence} (384-dimensional embeddings) pre-computed offline and stored in an in-memory FAISS index~\cite{johnson2019faiss}. At inference time, the patient query is encoded on-the-fly, and the top-$m$ most similar physician adapters are retrieved. For identity-based routing, this step is bypassed entirely, and the system directly loads the prescribed adapter.

\textbf{Stage 4: Inference with Merged Adapters.} We leverage LoRAX~\cite{lorax2024}, an open-source multi-LoRA inference server, for dynamic adapter loading and swapping at inference time. LoRAX enables hot-swapping of LoRA adapters without reloading the base model, supporting two deployment modes: (a) per-adapter serving, where each physician's LoRA adapter is loaded on demand via LoRAX and applied to the base model for generation (suitable for low-traffic scenarios or A/B testing); and (b) merged-adapter serving, where SCF is applied offline to produce a single merged adapter that is pre-loaded, and generation is conditioned on a physician-identifying prompt prefix. The merged mode supports a prompt template: ``\textit{string[Department: \{dept\}\string] \string[Physician ID: \{doc\_id\}\string] Patient: \{query\}$\backslash$Doctor:}'' that cues the model to generate in the appropriate personalized style.

\textbf{Inference Optimization.} We use FlashAttention-2 (xformers 0.0.31) for efficient attention computation. For memory-constrained deployments, QLoRA~\cite{dettmers2023qlora} can be employed to quantize the base model to 4-bit without significant performance degradation. Inference is served via vLLM 0.10.0, achieving an end-to-end latency of 1.2--2.8 seconds per response on a single H20-3e GPU, which meets the real-time requirements of internet hospital consultation workflows.
