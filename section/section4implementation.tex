\section{Implementation}

\subsection{Dataset Construction: IHDPD}

We construct the \textbf{Internet Hospital Doctor-Patient Dialogue (IHDPD)} dataset from the consultation records of a real-world internet hospital platform. The data collection and processing pipeline adheres to strict ethical and privacy protocols approved by the institutional review board.

\textbf{Data Source.} The raw data consist of text-based consultation records collected over a 24-month period (January 2024 to December 2025) from an operational internet hospital serving over 100,000 registered patients and more than 500 licensed physicians across 15 clinical departments, including internal medicine, cardiology, dermatology, pediatrics, obstetrics and gynecology, orthopedics, neurology, gastroenterology, endocrinology, psychiatry, ophthalmology, otolaryngology, urology, respiratory medicine, and general surgery.

\textbf{De-identification.} All personally identifiable information (PII) and protected health information (PHI) are systematically removed using a hybrid approach combining rule-based pattern matching (for structured identifiers such as names, ID numbers, phone numbers, and addresses) with a fine-tuned named entity recognition model~\cite{deidentification} (for contextual identifiers such as hospital names, dates, and locations). The de-identification pipeline achieves a recall of 98.7\% and a precision of 97.2\% on a manually annotated validation set of 500 dialogues. All de-identified records undergo a secondary manual review by two independent annotators to ensure completeness.

\textbf{Dialogue Structure.} Each consultation session in IHDPD is structured as a multi-turn dialogue between one patient and one physician. A typical session comprises an initial symptom description by the patient, followed by one or more rounds of physician inquiry, diagnostic reasoning, and treatment recommendation. The dataset preserves the natural turn-taking structure, including timestamps (with dates generalized to month-level granularity), department labels, and physician specialty annotations.

\textbf{Data Partitioning.} For each physician $d_k$ with at least 50 consultation records, we split their dialogues into training (70\%), validation (15\%), and test (15\%) sets. Physicians with fewer than 50 records are excluded from the personalization experiments but are retained in the dataset for aggregate baseline training. The final dataset used for personalization experiments comprises $K = 120$ physicians with a median of 186 dialogues per physician (range: 50--1,247).

\subsection{System Architecture}

Our system is implemented as a modular pipeline comprising four stages:

\textbf{Stage 1: Data Ingestion and Preprocessing.} Raw consultation records are streamed through the de-identification pipeline, tokenized using the base model's tokenizer, and formatted into instruction-following templates that include the patient query, physician department, and expected response format. The preprocessing stage is implemented using the weclone data utilities~\cite{weclone}, which handle batching, shuffling, and dynamic padding.

\textbf{Stage 2: LoRA Training.} We use the weclone training pipeline configured with the following hyperparameters: LoRA rank $r = 8$, LoRA alpha $\alpha = 16$, dropout rate $0.1$, target modules $\{q\_proj, v\_proj, k\_proj, o\_proj\}$, learning rate $2 \times 10^{-4}$ with cosine scheduling, batch size 4 with gradient accumulation over 8 steps (effective batch size 32), and a maximum of 10 training epochs with early stopping (patience = 3). Training is conducted on 4 NVIDIA A100-80GB GPUs using DeepSpeed ZeRO Stage-2 optimization.

\textbf{Stage 3: DAR Router.} The router encodes physician style descriptions $s_k$ using a distilled Sentence-BERT model~\cite{llm-eval} (384-dimensional embeddings) pre-computed offline and stored in an in-memory FAISS index~\cite{lag2024}. At inference time, the patient query is encoded on-the-fly, and the top-$m$ most similar physician adapters are retrieved. For identity-based routing, this step is bypassed entirely, and the system directly loads the prescribed adapter.

\textbf{Stage 4: Inference with Merged Adapters.} The system supports two deployment modes: (a) per-adapter serving, where each physician's LoRA adapter is loaded on demand and applied to the base model for generation (suitable for low-traffic scenarios or A/B testing); and (b) merged-adapter serving, where SCF is applied offline to produce a single merged adapter that is pre-loaded, and generation is conditioned on a physician-identifying prompt prefix. The merged mode supports a prompt template: ``\texttt{\\string[Department: \{dept\}\string] \string[Physician ID: \{doc\_id\}\string] Patient: \{query\}$\backslash$nDoctor:}'' that cues the model to generate in the appropriate personalized style.

\textbf{Inference Optimization.} We apply 4-bit quantization (NF4) to the base model and use FlashAttention-2 for efficient attention computation, achieving an end-to-end inference latency of 1.2--2.8 seconds per response on a single A100 GPU, which meets the real-time requirements of internet hospital consultation workflows.
