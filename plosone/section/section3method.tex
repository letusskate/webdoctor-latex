\section{Method}\label{section:design}

In this section, we present our three-stage personalized adaptation framework, illustrated in Figure~\ref{fig:method-overview}. We begin with a formal problem formulation, then describe each component in detail: doctor-specific LoRA training via LLaMA-Factory~\cite{zheng2024llamafactory}, Doctor-Adaptive Routing (DAR) for dynamic adapter routing, and SCF for parameter fusion.

\subsection{Problem Formulation}

Let $\mathcal{D} = \{d_1, d_2, \ldots, d_K\}$ denote the set of $K$ physicians in an internet hospital, and let $\mathcal{M}_0$ denote a pre-trained base language model. Each physician $d_k$ is associated with a historical dialogue corpus $\mathcal{C}_k = \{(q_i^{(k)}, r_i^{(k)})\}_{i=1}^{N_k}$, where $q_i^{(k)}$ is a patient query and $r_i^{(k)}$ is the physician's corresponding response. Our objective is to construct a system that, given a new patient query $q$ directed to physician $d_k$, generates a candidate response $\hat{r}$ that (1) is clinically accurate, (2) reflects physician $d_k$'s personal linguistic style and departmental norms, and (3) complies with institutional regulations.

Formally, we seek to learn physician-specific adaptation parameters $\Delta \Theta_k$ such that:
\begin{equation}
\hat{r} = \mathcal{M}_{\Theta_0 + \Delta \Theta_k}(q), \quad \text{where } \Delta \Theta_k \text{ captures the personalization for doctor } d_k.
\end{equation}

The key challenges are: (a) $N_k$ is typically small (tens to hundreds of examples), making full fine-tuning infeasible; (b) $K$ may be large, requiring efficient multi-doctor serving; and (c) the personalization signal must be disentangled from general clinical competence encoded in $\mathcal{M}_0$.

\subsection{Doctor-Specific LoRA Training}

We adopt Low-Rank Adaptation (LoRA)~\cite{hu2021lora} as our parameter-efficient fine-tuning strategy. For each physician $d_k$, we train a LoRA adapter that modifies the attention projection matrices of $\mathcal{M}_0$. Specifically, for a pre-trained weight matrix $W \in \mathbb{R}^{d \times d}$, LoRA constrains its update to a low-rank decomposition:
\begin{equation}
W + \Delta W = W + B A,
\end{equation}
where $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times d}$ are trainable low-rank matrices with rank $r \ll d$. During training, only $A$ and $B$ are optimized, while $W$ remains frozen.

We adopt LoRA fine-tuning based on LLaMA-Factory~\cite{zheng2024llamafactory} (via WeClone~\cite{xmingXming521WeClone2026}), an efficient framework for parameter-efficient fine-tuning that provides optimized kernels for training and inference. For each physician $d_k$, the training objective is the standard causal language modeling loss:
\begin{equation}
\mathcal{L}_k = -\sum_{t=1}^{T} \log P(r_t^{(k)} \mid q^{(k)}, r_{<t}^{(k)}; \Theta_0 + \Delta \Theta_k).
\end{equation}

To prevent overfitting given the limited per-physician data, we apply the following regularization strategies: (1) a small LoRA rank $r \in \{4, 8, 16\}$; (2) dropout on the LoRA matrices with rate $\alpha = 0.1$; and (3) early stopping based on validation perplexity on a held-out subset of each physician's dialogues. Each trained adapter $\Delta \Theta_k$ is serialized to disk as a compact checkpoint (typically $<10$ MB), enabling efficient storage and loading.

\subsection{Doctor-Adaptive Routing (DAR)}

At inference time, the system must select the appropriate LoRA adapter for the target physician. Let $e_k \in \mathbb{R}^{384}$ denote a pre-computed embedding vector encoding physician $d_k$'s communication style profile (derived from their historical responses via Sentence-BERT~\cite{reimers2019sentence}). For a new patient query $q$, we compute its embedding $e_q$ using the same encoder. The DAR router defines a scoring function:

\begin{equation}
\text{score}(q, d_k) = \cos(e_q, e_k) = \frac{e_q^\top e_k}{\|e_q\| \cdot \|e_k\|}
\end{equation}

We support two routing modes:

\textbf{Identity-Based Routing.} When the target physician $d_{k^*}$ is known at query time (the standard internet hospital scenario), the router directly selects adapter $k^*$ with zero computational overhead:
\begin{equation}
k_{\text{selected}} = k^*
\end{equation}

\textbf{Query-Based Routing.} When the optimal physician is not predetermined (e.g., triage or automated preliminary response generation), the router retrieves the top-$m$ most similar physicians by cosine similarity to the query embedding:
\begin{equation}
\mathcal{K}_m(q) = \underset{k \in \{1,\ldots,K\}}{\text{top-}m}\; \cos(e_q, e_k)
\end{equation}
The selected adapter(s) are then applied to the base model for generation. In practice, the FAISS index~\cite{johnson2019faiss} enables sub-millisecond nearest-neighbor retrieval over the full set of $K$ physician embeddings.

\textbf{Routing Context Augmentation.} Beyond the physician identity and query-text signals described above, the DAR router integrates four additional knowledge sources to enrich the routing context:
\begin{enumerate}
\item \textbf{Department-level regulations.} Each clinical department maintains codified rules governing permissible response content, specialty-specific disclaimers, and mandatory referral triggers. These regulations are stored as structured text snippets and retrieved by department ID at routing time.
\item \textbf{Hospital-wide institutional policies.} Global policies---including patient privacy disclaimers, escalation protocols, and medication prescription guidelines---are maintained as a centralized document set and prepended to the routing context regardless of department.
\item \textbf{Physician--patient dialogue history.} For returning patients, the most recent $N$ turns of the doctor--patient conversation are retrieved to provide continuity of care context, ensuring the generated response is clinically consistent with prior interactions.
\item \textbf{Physician-curated knowledge base.} Each physician maintains a personal editable archive of clinical notes, treatment templates, frequently-used drug combinations, and specialty references. These documents are indexed via a lightweight retrieval-augmented generation (RAG) pipeline built on FAISS~\cite{johnson2019faiss} with Sentence-BERT embeddings. At inference time, the top-$t$ most relevant knowledge snippets are retrieved and injected into the routing prompt, allowing physicians to dynamically refine their knowledge base without retraining the underlying LoRA adapter. Updates to the knowledge base propagate to the RAG index in real time.
\end{enumerate}
The augmented routing context $C_k$ for physician $d_k$ and query $q$ is assembled as: $C_k = \{ \text{dept\_rules}, \text{hospital\_policy}, \text{history}_{N}, \text{RAG\_top}_{t} \}$. This context is prepended to the prompt template before generation, ensuring all regulatory, institutional, and personal knowledge constraints are satisfied simultaneously.

\subsection{SCF for Parameter Fusion}

While per-physician LoRA adapters provide maximal personalization, serving $K$ separate adapters incurs linear storage and loading costs. SCF fuses multiple adapters into a single merged artifact while minimizing interference~\cite{yadavTIESMergingResolvingInterference2023a}. Formally, let $\Delta_k \in \mathbb{R}^{d}$ denote the flattened LoRA parameter delta for physician $d_k$. SCF proceeds in three stages:

\textbf{Stage 1: Trim.} Low-magnitude parameter changes likely representing noise are reset to zero. For each parameter dimension $j$, we retain only the top $(1-\tau)$ fraction of parameters by magnitude:
\begin{equation}
\Delta_{k,j}^{(1)} = \begin{cases}
\Delta_{k,j}, & \text{if } |\Delta_{k,j}| \ge \text{Q}_{\tau}(\{|\Delta_{1,j}|,\ldots,|\Delta_{K,j}|\}) \\
0, & \text{otherwise}
\end{cases}
\end{equation}

\textbf{Stage 2: Elect Sign.} For each dimension, we resolve sign conflicts via a weighted majority vote with soft sign parameter $\eta \ge 0$:
\begin{equation}
s_j^* = \operatorname{sign}\!\left(\sum_{k=1}^{K} |\Delta_{k,j}^{(1)}|^{\eta} \cdot \operatorname{sign}(\Delta_{k,j}^{(1)})\right)
\end{equation}
When $\eta=1$, all deltas contribute equally; as $\eta \to \infty$, only the largest-magnitude delta dominates.

\textbf{Stage 3: Merge.} Only parameters whose sign agrees with $s_j^*$ are merged. An adaptive trim coefficient $\lambda \in [0,1]$ controls merging aggressiveness:
\begin{equation}
\Delta_j^{\text{(merged)}} = \frac{1}{|\mathcal{A}_j|} \sum_{k \in \mathcal{A}_j} \Delta_{k,j}^{(1)},\quad
\mathcal{A}_j = \{k : \operatorname{sign}(\Delta_{k,j}^{(1)}) = s_j^*\}
\end{equation}

The merged adapter is applied as $\Theta^{\text{(merged)}} = \Theta_0 + \Delta^{\text{(merged)}}$. Default hyperparameters are $\eta=1.0$, $\lambda=0.5$, and $\tau=0.20$, with sensitivity analyzed in Section~\ref{section:evaluation}.
